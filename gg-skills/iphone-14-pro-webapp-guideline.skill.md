# iPhone 14 Pro WebApp Design Guideline

**Permanent constraint for all Terrence projects.** iPhone 14 Pro (393 × 852 CSS points, DPR 3) is the **sole design baseline** for every frontend generated by Crush/OpenCode. Never design for desktop-first or Android-first. Never ask "which device?" — the answer is always iPhone 14 Pro.

## When to load this skill

This skill must be loaded when generating **any frontend code, CSS, HTML, or WebApp UI** for Terrence. Trigger keywords:

- iPhone 14 Pro

## Core Baseline

| Property | Value |
|---|---|
| CSS Viewport | **393 × 852** CSS points |
| Device Pixel Ratio | **3×** (1 CSS point = 3 physical pixels) |
| Physical Resolution | 1179 × 2556 px |
| Safe Area Top | **59 pt** (notch + status bar) |
| Safe Area Bottom | **34 pt** (Home indicator) |
| Safe Area Sides | **0 pt** (edge-to-edge display) |
| Status Bar Height | **54 pt** |
| Navigation Bar Height | **44 pt** |
| Tab Bar Height | **49 pt** (plus safe area bottom) |
| Minimum Touch Target | **44 × 44 pt** (Apple HIG) |
| Content Area (no bars) | 852 - 59(top) - 34(bottom) = **759 pt** |
| Content Area (tab bar) | 852 - 59 - 49 - 34 = **710 pt** |
| Content Area (nav + tab) | 852 - 44(nav) - 49(tab) - 59 - 34 = **666 pt** |
| Font Size Minimum | **17 pt** body text (iOS default) |
| Tap Highlight | Use `-webkit-tap-highlight-color: transparent` with custom active states |

## Viewport Meta Configuration

```html
<meta name="viewport" content="width=device-width, initial-scale=1.0, maximum-scale=1.0, user-scalable=no, viewport-fit=cover" />
```

- `viewport-fit=cover` — essential for edge-to-edge display on iPhone X+ devices
- `user-scalable=no` — prevents accidental zoom on form inputs (standard for WebApps)

## WebApp Config (add-to-home-screen)

```html
<!-- iOS Safari WebApp meta tags -->
<meta name="apple-mobile-web-app-capable" content="yes" />
<meta name="apple-mobile-web-app-status-bar-style" content="default" />
<!-- translucent (blur) | black | default -->
<meta name="apple-mobile-web-app-title" content="App Name" />
<link rel="apple-touch-icon" sizes="180x180" href="icon-180.png" />
<link rel="apple-touch-startup-image" href="splash.png" />
```

## CSS Safe Area Variables

Always use `env(safe-area-inset-*)` for layout padding. Never hardcode safe-area values.

```css
:root {
  --sat: env(safe-area-inset-top);    /* 59px on iPhone 14 Pro */
  --sar: env(safe-area-inset-right);   /* 0px */
  --sab: env(safe-area-inset-bottom);  /* 34px on iPhone 14 Pro */
  --sal: env(safe-area-inset-left);    /* 0px */
  --safe-top: var(--sat);
  --safe-bottom: var(--sab);
}
```

When using `position: fixed` for headers or footers:

```css
.fixed-header {
  position: fixed;
  top: 0;
  left: 0;
  right: 0;
  padding-top: var(--sat);
  height: calc(44px + var(--sat));
}

.fixed-footer {
  position: fixed;
  bottom: 0;
  left: 0;
  right: 0;
  padding-bottom: var(--sab);
  height: calc(49px + var(--sab));
}
```

## Touch Target Requirements (Apple HIG)

- **Minimum: 44 × 44 pt** for all interactive elements
- **Spacing:** 8 pt minimum between touch targets
- **Visual targets smaller than 44 pt** must have invisible padding to meet 44 pt hit area
- **Buttons with icons** should have a hit area larger than the icon itself

```css
/* Good — 44pt minimum touch target */
.button {
  min-width: 44px;
  min-height: 44px;
  padding: 12px 16px; /* ensures hit area */
}

/* Icon buttons need invisible padding */
.icon-button {
  width: 24px;        /* visual size */
  height: 24px;
  padding: 10px;      /* expands to 44×44 hit area */
  /* or use: min-width: 44px; min-height: 44px; */
}
```

## Typography Guidelines

| Role | Size | Weight | Line Height | Usage |
|---|---|---|---|---|
| Large Title | 34 pt | Bold (700) | 41 pt | Hero headers |
| Title 1 | 28 pt | Bold (700) | 34 pt | Screen titles |
| Title 2 | 22 pt | Bold (700) | 28 pt | Section headers |
| Title 3 | 20 pt | Semibold (600) | 25 pt | Sub-section headers |
| Headline | 17 pt | Semibold (600) | 22 pt | Important labels |
| Body | 17 pt | Regular (400) | 22 pt | Primary reading text |
| Callout | 16 pt | Regular (400) | 21 pt | Captions below body |
| Subhead | 15 pt | Regular (400) | 20 pt | Secondary text |
| Footnote | 13 pt | Regular (400) | 18 pt | Tertiary / metadata |
| Caption 1 | 12 pt | Regular (400) | 16 pt | Small labels / timestamps |
| Caption 2 | 11 pt | Regular (400) | 13 pt | Legal / smallest text |

**Line lengths:** Aim for 40-60 characters per line. On 393 pt width with 20 pt side padding, you get ~353 pt usable. At 17 pt body, that is ~20-21 characters per line — within range for comfortable reading.

## Spacing System

Base unit: 8 pt multiples.

| Name | Value | Usage |
|---|---|---|
| Spacing XXS | 4 pt | Smallest gaps |
| Spacing XS | 8 pt | Between related elements |
| Spacing SM | 12 pt | Between grouped elements |
| Spacing MD | 16 pt | Standard content padding |
| Spacing LG | 24 pt | Section spacing |
| Spacing XL | 32 pt | Major section breaks |
| Spacing XXL | 48 pt | Page-level sections |
| Content Padding | 16 pt or 20 pt | Horizontal edge padding |
| Grid Gutter | 16 pt | Between grid columns |

## Layout Grid

- **Total width:** 393 pt
- **Content padding:** 16 pt each side (total 32 pt)
- **Content area:** 361 pt
- **Columns:** 4-column grid (mobile standard)
  - Column width: ~78 pt
  - Gutter: 16 pt
- **Alternative:** 2-column grid for list/detail layouts

```css
.grid-4 {
  display: grid;
  grid-template-columns: repeat(4, 1fr);
  gap: 16px;
  padding: 0 16px;
}
```

## iOS Safari WebView / WKWebView Behaviors

### Keyboard Handling
- iOS keyboard pushes the viewport up (does NOT resize window)
- Use `visualViewport` API for keyboard-aware layouts
- Fixed elements reposition when keyboard opens
- Input fields should scroll into view when focused

```javascript
if (window.visualViewport) {
  window.visualViewport.addEventListener('resize', () => {
    // Handle keyboard appearing/disappearing
    const keyboardHeight = window.visualViewport.height - document.documentElement.clientHeight;
    if (keyboardHeight > 0) {
      document.activeElement?.scrollIntoView({ block: 'center' });
    }
  });
}
```

### Scrolling
- Use `-webkit-overflow-scrolling: touch` for smooth scroll containers
- **Do not** use `overflow: scroll` on `<body>` — use a content wrapper
- Pull-to-refresh is native to Safari — decide if you want to override via `overscroll-behavior`

### Text Size
- iOS Safari auto-adjusts font size at certain widths / orientations
- Prevent with `text-size-adjust: 100%` in CSS
- 17 pt minimum body text is a requirement for WebApps

### Link / Call Styles
- `tel:` links auto-detect and style as phone numbers
- `data:` URIs in links may cause issues
- Use `-webkit-touch-callout: none` to disable callout on long-press

## Input / Form Styling

```css
/* Reset iOS default input styling */
input, textarea, select {
  -webkit-appearance: none;
  appearance: none;
  border-radius: 10px; /* iOS-style */
  font-size: 17px; /* prevents iOS zoom-on-focus */
}

/* Prevent zoom on focus — font-size must be ≥ 16px */
input, textarea, select {
  font-size: 17px;
}
```

## Status Bar & Notch

- Status bar area: 54 pt (notch region)
- Top safe area: 59 pt (status bar + rounded corners buffer)
- Content should never overlap the notch / Dynamic Island area
- Use `viewport-fit=cover` + safe-area padding, never fixed-position top content without padding

## Color & Visual Design

- Support Dark Mode via `prefers-color-scheme: dark`
- Use system colors where appropriate: `-webkit-system-colors`
- Contrast: WCAG AA minimum (4.5:1 for text, 3:1 for large text)
- Backdrop blur works well for navigation bars and sheets

```css
.nav-bar {
  background: rgba(255, 255, 255, 0.72);
  -webkit-backdrop-filter: saturate(180%) blur(20px);
  backdrop-filter: saturate(180%) blur(20px);
}

@media (prefers-color-scheme: dark) {
  .nav-bar {
    background: rgba(0, 0, 0, 0.72);
  }
}
```

## Orientation Support

- iPhone 14 Pro supports portrait and landscape
- Landscape on 393 × 852 swaps to 852 × 393
- Safe areas change: top/bottom become side safe areas in landscape
- Tab bars often hide in landscape for content density
- Always test both orientations

## Permanent Constraint Enforcement

This is a **permanent constraint**, not a guideline that can be overridden:

1. **Every frontend** for Terrence targets this baseline
2. **Every CSS rule** must use safe-area-aware variables
3. **Every touch target** must meet 44×44 pt minimum
4. **Every viewport** must use `viewport-fit=cover`
5. **Never** generate code targeting other devices unless explicitly told to add responsive support beyond this baseline
6. **Never** use hardcoded pixel values for layout spacing — use the spacing system
7. **Every input** must have `font-size: 17px` minimum to prevent iOS zoom
8. **Default font** should be the iOS system font: `-apple-system, BlinkMacSystemFont, 'SF Pro Text', 'Helvetica Neue', sans-serif`
