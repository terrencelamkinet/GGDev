## 一、核心安全理念

### 1.1 信息安全定義（ISO）
> 「為數據處理系統建立和採取技術、管理的安全保護，保護計算機硬件、軟件、數據不因偶然的或惡意的原因而受到破壞、更改、泄露」

### 1.2 CIA三元組（基本屬性）
| 屬性 | 定義 | 檢查問題 |
|------|------|----------|
| **保密性** Confidentiality | 防止未授權的信息泄露 | 誰可以看到？是否加密？ |
| **完整性** Integrity | 防止未授權的修改或破壞 | 數據是否被篡改？Checksum？ |
| **可用性** Availability | 確保系統可被授權者使用 | 冗餘？備份？恢復時間？ |

### 1.3 其他關鍵屬性
- 真實性 Authenticity
- 不可否認性 Non-repudiation
- 可問責性 Accountability
- 可靠性 Reliability

### 1.4 信息安全問題根源
- **內因**：信息系統複雜性導致漏洞不可避免
- **外因**：環境因素 + 人為因素

### 1.5 信息安全特徵
- **系統性** — 不能只看單點，要整體考慮
- **動態性** — 威脅不斷變化，防禦需持續更新
- **無邊界** — 雲端、移動、IoT 已打破傳統邊界
- **非傳統** — 不只是技術問題，還涉及管理、人、流程

## 二、安全保障框架

### 2.1 信息安全發展階段
1. 通信安全階段（40-70年代）— 加密為主
2. 計算機安全階段（70-90年代）— 訪問控制
3. 信息系統安全階段（90年代後）— 防火牆+IDS+VPN
4. **信息安全保障階段**（1996+）— 技術+管理+工程+人員綜合保障
5. 網絡空間安全階段 — 防禦+威懾+利用三位一體

### 2.2 我國信息安全保障
- **總體要求**：積極防禦，綜合防範
- **主要原則**：技術與管理並重，正確處理安全與發展的關係

### 2.3 威脅情報與態勢感知
- 威脅情報 = 為管理決策提供依據
- 態勢感知 = 在威脅情報基礎上，通過大數據分析預判未來威脅狀態

## 三、風險管理（核心思想）

> **基於風險的思想是所有信息系統安全保障工作的核心思想**

### 3.1 風險管理基本過程（GB/Z 24364）

```
背景建立 → 風險評估 → 風險處理 → 批准監督
     ↕ 監控審查 ↕        ↕ 溝通諮詢 ↕
```

#### Step 1: 背景建立
- 確定風險管理對象和範圍
- 組建團隊、制定計劃、獲得支持
- 信息系統調查（業務目標、技術特點）
- 信息系統分析（體系結構、關鍵要素）
- 信息安全分析（安全要求、安全環境）

#### Step 2: 風險評估
- 制定評估方案、選擇方法
- 識別威脅、脆弱性、控制措施
- 分析風險可能性與影響
- 判定風險等級

#### Step 3: 風險處理
- 判斷哪些風險可接受
- 確認處理目標
- 選擇處理方式（規避、轉移、降低、接受）
- 實施控制措施

#### Step 4: 批准監督
- 管理層認可風險管理結果
- 監督環境變化是否引入新風險

#### 監控審查 + 溝通諮詢（貫穿全過程）

### 3.2 常見風險管理模型
- **COSO報告** — 內部控制整合框架（3目標5要素）
- **ISO 31000** — 風險管理最佳實踐
- **COBIT** — 信息系統治理及控制

### 3.3 風險處理選項
| 方式 | 說明 | 適用場景 |
|------|------|----------|
| 規避 | 停止活動 | 風險太高無可控 |
| 轉移 | 買保險、外包 | 低概率高影響 |
| 降低 | 加控制措施 | 最常見 |
| 接受 | 不做處理 | 風險低於接受線 |

### 4.2 PDCA 過程方法
```
P (Plan)    — 建立：組織背景、領導力、計劃、支持
D (Do)      — 實施：風險評估、控制措施選取、部署
C (Check)   — 檢查：監控績效、評審
A (Act)     — 行動：不符合糾正、持續改進
```

### 4.3 ISMS 成功關鍵因素
1. 安全策略、目標與活動一致
2. 與組織文化匹配的方法與框架
3. 最高管理層的可見支持與承諾
4. 基於風險管理的資產保護
5. 有效的安全意識培訓
6. 有效的事件管理過程
7. 有效的業務連續性管理
8. 測量體系 + 反饋改進

## 五、安全評估

### 5.1 評估工作內容
1. 確定保護對象（資產）是什麼？其直接和間接價值？
2. 資產面臨哪些潛在威脅？
3. 導致威脅的問題所在？
4. 現有控制措施是否足夠？
5. 剩餘風險是否可接受？

### 5.2 評估方法
- **檢查表法**：按標準清單逐項檢查
- **問卷調查**：收集人員認知
- **訪談**：深入理解管理現狀
- **漏洞掃描**：技術層面自動檢測
- **滲透測試**：模擬攻擊者視角
- **審計**：系統性獨立審查

### 5.3 審計重點
- 是否滿足合規要求
- 文件與實際執行是否一致
- 控制措施是否有效運作
- 技術符合性核查

## 六、安全工程與運營

### 6.1 系統安全工程
- 採用工程概念、原理、技術來開發維護安全的系統
- 良好安全工程四方面：**策略、機制、保證、動機**
- 成熟度模型：從混亂到優化（CMMI 概念）

### 6.2 安全運營關鍵控制
| 領域 | 關鍵措施 |
|------|----------|
| 操作規程 | 文件化、變更管理、容量管理、環境分離 |
| 惡意代碼防範 | 防病毒、軟件白名單 |
| 備份 | 定期備份、離線備份、恢復演練 |
| 日誌監控 | 事件日誌、管理員日誌、時鐘同步 |
| 漏洞管理 | 定期掃描、及時修補、安裝限制 |
| 審計考量 | 審計活動不影響業務 |

### 6.3 業務連續性管理
- 信息安全連續性嵌入業務連續性管理
- 冗餘確保可用性
- 定期確認、審查、評估

## 七、安全支撐技術

### 7.1 密碼學
- 對稱密鑰：加解密速度快，密鑰分發是問題（AES）
- 非對稱密鑰：用公鑰加密私鑰解密（RSA、ECC）
- 哈希：完整性驗證（SHA-256）
- **科克霍夫準則**：密碼系統的安全性應依賴於密鑰的保密，而非算法的保密

### 7.2 訪問控制
- **DAC**（自主訪問控制）— 用戶自行決定權限
- **MAC**（強制訪問控制）— 系統強制標籤
- **RBAC**（基於角色）— 按角色分配權限
- 最小權限原則
- 職責分離

### 7.3 身份鑑別
- 你知道的（口令）
- 你擁有的（Token、手機）
- 你是誰（指紋、人面）
- 多因素認證（MFA）= 兩種以上組合

## 十、應用於三台 Server 的檢查清單

基於 CISP 框架，每次風險評估需覆蓋：

### 基礎檢查
- [ ] 系統版本及補丁狀態
- [ ] SSH 配置（密碼/密鑰、端口、root 登入）
- [ ] 防火牆規則（最小開放原則）
- [ ] 用戶與權限（sudoers、非 root 日常使用）
- [ ] 敏感信息存儲（API Key、密碼、憑證）

### 訪問控制
- [ ] 是否有未授權用戶？
- [ ] 特權帳號是否共享？
- [ ] MFA 是否啟用？
- [ ] 最小權限原則是否落實？

### 日誌與監控
- [ ] 系統日誌是否啟用？
- [ ] 日誌是否集中管理？
- [ ] 異常是否告警？

### 備份與恢復
- [ ] 關鍵數據是否備份？
- [ ] 備份是否離線/異地？
- [ ] 恢復演練是否定期進行？

### 網絡安全
- [ ] 不必要的端口是否關閉？
- [ ] 服務是否最小化安裝？
- [ ] 是否有網絡隔離？

### 漏洞管理
- [ ] 上次漏洞掃描時間？
- [ ] 已知 CVE 是否已處理？
- [ ] 第三方依賴是否定期審查？

## 十一、持續改進機制

- 每次風險評估後更新檢查清單
- 定期審計（建議每季）
- 重大變更後重新評估
- 事件發生後進行 Root Cause 分析
- 將經驗沉澱為可重用的 Skill
