# Patterns — AI Agent Instruction Spec
> Reference: Apple HIG — Patterns
> Covers: Navigation, Searching, Onboarding, Settings, Feedback, Loading, Modality, Permissions, Data Entry, Undo & Redo, Drag & Drop, File Management, Collaboration, Handoff, Notifications, Printing, Quick Actions, Ratings & Reviews, Workouts & Health

---

## 1. Navigation

### Navigation Models
| Model | Description | Use When |
|-------|-------------|----------|
| **Hierarchical** | One choice per screen, drill down | Settings, Mail, Files |
| **Flat (Tab-based)** | Switch between parallel sections | App Store, Music, Instagram |
| **Content-Driven** | Free-form movement through content | Games, Books, Safari |

### AI Agent Rules
| Rule ID | Instruction |
|---------|-------------|
| NAV-001 | Choose one primary navigation model — do not combine hierarchical + flat at the same level. |
| NAV-002 | Tab bar: maximum **5 tabs**; use sidebar for more (iPad/Mac). |
| NAV-003 | Navigation bar title: use `.large` title for root screens, `.inline` for drill-down screens. |
| NAV-004 | Back button: always auto-generated by `NavigationStack` — never create a custom back button. |
| NAV-005 | Sheets: use for focused tasks that don't require full navigation; provide explicit dismiss action. |
| NAV-006 | Full-screen modals: use only for immersive tasks (camera, media playback). |
| NAV-007 | Never nest more than **3 levels deep** without providing a clear way to jump back. |
| NAV-008 | Deep links must restore exact navigation state (not just root screen). |

## 2. Searching

### Search UI Anatomy
```
[ 🔍 Search placeholder... ] [Cancel]
  ┌─────────────────────────────┐
  │  Recent Searches            │
  │  Suggestions / Tokens       │
  │  Results List               │
  └─────────────────────────────┘
```

### AI Agent Rules
| Rule ID | Instruction |
|---------|-------------|
| SCH-001 | Use `UISearchController` / `.searchable()` modifier — not a custom text field. |
| SCH-002 | Show search suggestions as the user types (minimum 1 character). |
| SCH-003 | Show "No Results" state with a helpful suggestion — never an empty blank screen. |
| SCH-004 | Persist recent searches locally; allow individual or bulk clear. |
| SCH-005 | Search scope bar: use when content has distinct filterable categories. |
| SCH-006 | Search must be triggered on each keystroke (live results), not only on submit. |

## 3. Onboarding

### AI Agent Rules
| Rule ID | Instruction |
|---------|-------------|
| ONB-001 | Show onboarding only once — not on every launch. |
| ONB-002 | Maximum **3–5 onboarding screens**; each with a single, clear message. |
| ONB-003 | Always provide a **Skip** option. |
| ONB-004 | Do NOT request permissions during onboarding — wait until the feature is first used. |
| ONB-005 | Launch screen (splash): system-generated from Storyboard/Info.plist — never a static image with fake loading. |
| ONB-006 | Use `WHATsNew` pattern (feature highlights) for major app updates — not full onboarding. |
| ONB-007 | First launch should show the user their content as quickly as possible. |

## 4. Settings

### AI Agent Rules
| Rule ID | Instruction |
|---------|-------------|
| SET-001 | Simple settings (toggles, selections): embed in-app in a Settings screen. |
| SET-002 | System-level settings (notifications, permissions): link to system Settings app via `UIApplication.openSettingsURLString`. |
| SET-003 | Never require settings configuration before the app can be used. |
| SET-004 | Group related settings into sections with descriptive headers. |
| SET-005 | Destructive settings (delete all data, sign out) must be at the bottom, colored red, and require confirmation. |

## 5. Feedback (User Feedback)

### Feedback Types
| Type | Mechanism | Use Case |
|------|-----------|----------|
| Visual | State change, color, animation | Button tap confirmation |
| Haptic | `UIImpactFeedbackGenerator` | Selection, completion, error |
| Audio | System sounds or custom | Notifications, game events |
| Alert | `UIAlertController` | Errors requiring action |
| Toast/Banner | Transient notification | Success confirmations |

### AI Agent Rules
| Rule ID | Instruction |
|---------|-------------|
| FBK-001 | Every user action must produce **immediate visual feedback** (≤100ms response). |
| FBK-002 | Use haptics for: selection changes, task completion, errors, and destructive confirmations. |
| FBK-003 | Error messages must tell the user **what went wrong** and **what to do next**. |
| FBK-004 | Never use error codes in user-facing messages — always use plain language. |
| FBK-005 | Success states: use `.systemGreen` confirmation with a checkmark icon + brief message. |

## 6. Loading States

### AI Agent Rules
| Rule ID | Instruction |
|---------|-------------|
| LOD-001 | Always show a loading indicator when content fetch exceeds **0.5 seconds**. |
| LOD-002 | Use skeleton screens (placeholder shapes) for content-heavy lists and feeds. |
| LOD-003 | Use `ProgressView` (spinner) for indeterminate loading; `ProgressView(value:total:)` for determinate. |
| LOD-004 | Never show an empty screen without context — explain why content is missing. |
| LOD-005 | Cache previously loaded content so users see stale-but-useful data while refreshing. |
| LOD-006 | Provide pull-to-refresh for list-based content. |

## 7. Modality

### AI Agent Rules
| Rule ID | Instruction |
|---------|-------------|
| MOD-001 | Use modals **sparingly** — only when the user must complete a task before continuing. |
| MOD-002 | Every modal must have an explicit dismiss path (Cancel button, close icon, or swipe-to-dismiss). |
| MOD-003 | Alerts: use only for critical information requiring immediate decision. Maximum 2 action buttons. |
| MOD-004 | Action Sheets: use for offering multiple choices related to a specific action. |
| MOD-005 | Sheet detents: use `.medium` + `.large` for resizable sheets (iOS 16+). |
| MOD-006 | Never stack multiple modals — resolve the current modal before presenting another. |

## 8. Requesting Permissions

### AI Agent Rules
| Rule ID | Instruction |
|---------|-------------|
| PRM-001 | Request permission **at the moment the feature is first used** — not at app launch. |
| PRM-002 | Provide a **pre-permission alert** explaining why the permission is needed before the system prompt. |
| PRM-003 | If denied, gracefully degrade or offer an alternative path. Never block the entire app. |
| PRM-004 | Permissions to include purpose strings: Camera, Microphone, Location, Contacts, Photos, Health, Notifications. |
| PRM-005 | Check permission status on every app launch — never assume it remains granted. |

## 9. Data Entry

### AI Agent Rules
| Rule ID | Instruction |
|---------|-------------|
| DAT-001 | Use the appropriate keyboard type for each input field (`numberPad`, `emailAddress`, `phonePad`, `URL`). |
| DAT-002 | Show inline validation as the user types — do not wait until form submission. |
| DAT-003 | Pre-fill fields with known data (name, address from Contacts if permitted). |
| DAT-004 | Provide a "Next" keyboard action button to advance through form fields in sequence. |
| DAT-005 | Auto-capitalize names/sentences; disable for email, URL, passwords. |
| DAT-006 | Mask password fields; provide a "show/hide" toggle. |
| DAT-007 | Support AutoFill for passwords (`textContentType = .password`) and one-time codes. |

## 10. Undo & Redo

### AI Agent Rules
| Rule ID | Instruction |
|---------|-------------|
| UND-001 | Support shake-to-undo for text editing via `UIUndoManager`. |
| UND-002 | macOS: always implement `Cmd+Z` (undo) and `Cmd+Shift+Z` (redo). |
| UND-003 | Destructive actions (delete, archive) must offer an **undo** option — ideally via a snackbar/toast with countdown. |
| UND-004 | Undo history must persist across navigation within the same document/session. |

## 11. Drag & Drop

### AI Agent Rules
| Rule ID | Instruction |
|---------|-------------|
| DRG-001 | Support system drag-and-drop for all content items (images, text, files, rows). |
| DRG-002 | Provide visual lift animation when a drag begins — use system default spring animation. |
| DRG-003 | Show valid drop targets with highlight when item is dragged over them. |
| DRG-004 | Support multi-item drag (drag multiple selected items simultaneously). |

---

## 12. Notifications

### AI Agent Rules
| Rule ID | Instruction |
|---------|-------------|
| NOT-001 | Request notification permission only when there is clear value to the user. |
| NOT-002 | Notifications must be actionable — include a deep link to the relevant content. |
| NOT-003 | Support **Notification Actions** (interactive buttons in the notification) where appropriate. |
| NOT-004 | Group related notifications using `threadIdentifier`. |
| NOT-005 | Support **Live Activities** for ongoing real-time events (deliveries, rides, scores). |

---

## 13. Quick Actions (Home Screen)

### AI Agent Rules
| Rule ID | Instruction |
|---------|-------------|
| QCK-001 | Provide 3–4 Quick Actions on the Home Screen icon (long-press menu). |
| QCK-002 | Quick Actions should represent the most common entry points: New, Favorites, Search, Scan. |
| QCK-003 | Use SF Symbols for Quick Action icons — must be recognizable at small size. |

---

## 14. Ratings & Reviews

### AI Agent Rules
| Rule ID | Instruction |
|---------|-------------|
| RAT-001 | Use `SKStoreReviewController.requestReview()` — never build a custom rating prompt. |
| RAT-002 | Trigger review request after a **positive user moment** (task completed, milestone reached). |
| RAT-003 | Maximum 3 review prompts per 365-day period (system-enforced). |
| RAT-004 | Never gate features or content on leaving a review. |
